実は今月から、さくらインターネットの sacloud OSS 担当になりました。
というわけで、今月の sacloud OSS の更新情報をまとめてみました。

## terraform-provider-sakura

新世代の Terraform プロバイダーである `terraform-provider-sakura` は、4 月に 3 回のリリースがありました。モニタリングスイート関連と、新しいプロダクトへの対応が中心です。

### 主要な機能追加

- **モニタリングスイート対応の充実** が進みました。[v3.8.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.8.0) でストレージ・ルーティング・Dashboard リソースが追加され、[v3.8.1](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.8.1) では Alert リソース、[v3.9.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.9.0) では `retention_period_days` の設定にも対応しています。これでモニタリングスイートまわりは一通り Terraform で管理できる状態に近づきました。
- **AppRun 専有型に対応** しました ([v3.8.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.8.0))。AppRun 専有型のリソースを Terraform で扱えるようになっています。
- **Service Endpoint Gateway に対応** しました ([v3.9.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.9.0))。サービスエンドポイントゲートウェイ (SEG) は 2025 年 10 月に提供開始されたスイッチの拡張機能で、プライベートネットワーク内のサーバーからオブジェクトストレージや AI Engine などのマネージドサービスへ、インターネットを経由せずにプライベート接続できるようにするものです ([プロダクトページ](https://cloud.sakura.ad.jp/news/2025/10/30/service-endpoint-gateway/))。これが Terraform で管理できるようになりました。

### 主要なバグ修正・破壊的変更

- `monitoring_suite` のルーティングリソースで `resource_id` を optional にしました ([v3.8.1](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.8.1))。
- `container_registry` の `access_level` から `readwrite` を削除しました ([v3.9.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.9.0))。これは API 側の仕様変更に追従するもので、既存の設定で `readwrite` を指定していた場合は変更が必要です。
- `enhanced_db` リソースが `ondemand_db` にリネームされました ([v3.9.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.9.0))。プロダクト名の変更に伴うものです。

`container_registry` と `enhanced_db` (旧名) を使っている方は、アップデート時にお気をつけください。

https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.9.0

## usacloud

CLI クライアントの `usacloud` は [v1.22.0](https://github.com/sacloud/usacloud/releases/tag/v1.22.0) がリリースされました。

### 主要な機能追加

- **`bill` コマンドと `coupon` コマンドで auth context をサポート** するようになりました。これまで他のコマンドでは使えていた auth context の仕組みが、請求関連のコマンドにも適用されています。複数アカウントを切り替えて使う方には便利かと思います。

## その他の sacloud OSS

### IaaS API ライブラリ群

低レベル API の `iaas-api-go` と高レベル API の `iaas-service-go` がそれぞれ 2 本ずつリリースされました。

- `iaas-api-go` は [v1.27.0](https://github.com/sacloud/iaas-api-go/releases/tag/v1.27.0) で ContainerRegistry の public setting から `ReadWrite` が削除されました。続く [v1.28.0](https://github.com/sacloud/iaas-api-go/releases/tag/v1.28.0) では NFS plan queries に zone パラメータを渡せるようバグ修正が入っています。NFS のプラン情報を取りたいときに、ゾーン未指定で正しく動作しないケースがあったのを修正したものです。
- `iaas-service-go` の [v1.23.0](https://github.com/sacloud/iaas-service-go/releases/tag/v1.23.0) では `authstatus` パッケージが deprecated になりました。代替として `iam-api-go` の auth context API を利用するように移行する形になります。

### iam-api-go

`iam-api-go` の [v0.3.0](https://github.com/sacloud/iam-api-go/releases/tag/v0.3.0) で IAM API v1.3.0 に対応しました。利用者向けに新しく増えたものとしては、以下のあたりが目玉です。

- **ユーザープロビジョニング API** (`/scim-configurations`) が追加されました。SCIM ベースで外部 IdP からユーザーを管理できる API です。`apis/scim` パッケージとして使えます。
- **認証情報コンテキスト取得 API** (`GET /auth/context`) が追加されました。現在使っている API キーまたはサービスプリンシパルの ID・認証種別・操作可能なプロジェクト ID を取得できる API で、usacloud のような OSS プロダクトからの利用が想定されています。`apis/auth` の `GetAuthContext` から呼べます。
- IAM ロールのレスポンスボディに `lowest_grantable_resource` フィールドが追加されました。

### apprun-api-go

`apprun-api-go` の [v0.7.1](https://github.com/sacloud/apprun-api-go/releases/tag/v0.7.1) で AppRun の OpenAPI spec v1.3.0 に追従しました。今回のメインは、リソースを操作する API のレスポンスから `resource_id` を取得できるようになったことです。Terraform プロバイダー側でリソース ID を保持する流れに必要だったもので、`terraform-provider-sakura` [v3.8.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.8.0) の `apprun_shared` 改善とセットの変更です。

### saclient-go

低レベル HTTP クライアントの `saclient-go` も 4 月は 3 本のリリースがありました。利用者から見える変更を中心にいくつか紹介します。

- [v0.3.5](https://github.com/sacloud/saclient-go/releases/tag/v0.3.5) でトークン関連のバグを 2 件修正しました。トークンの有効期限スキューの方向が逆向きになっていた問題と、トークン取得前のロック昇格の問題です。長時間動かすアプリで認証エラーが出ていた方は、改善が見込めるかもしれません。
- [v0.3.6](https://github.com/sacloud/saclient-go/releases/tag/v0.3.6) では、`SAKURA_ACCEPT_LANGUAGE` 環境変数を指定したときにクラッシュする不具合を修正しました。
- [v0.3.7](https://github.com/sacloud/saclient-go/releases/tag/v0.3.7) で `IsConflictError` ヘルパーが追加されました。さくらのクラウドには「最初にアカウント相当を作る必要があるサービス」がいくつかあり、Terraform やテストで「すでに存在しているか」をチェックしづらい場面があります。とりあえず Create を投げてみて 409 (Conflict) かどうかで分岐するパターンを書きやすくするためのヘルパーです。あわせてエンドポイント設定で zone 指定が再サポートされました。

### sacloud-otel-collector

OpenTelemetry Collector のディストリビューションである `sacloud-otel-collector` は [v0.6.1](https://github.com/sacloud/sacloud-otel-collector/releases/tag/v0.6.1) がリリースされ、`groupbyattrs` processor が追加されました。属性でグループ化してから downstream に流したい場面で使えます。

### service-endpoint-gateway-api-go

新プロダクト Service Endpoint Gateway 用の API ライブラリです。今月、初版の [v0.1.0](https://github.com/sacloud/service-endpoint-gateway-api-go/releases/tag/v0.1.0) がリリースされ、続く [v0.1.1](https://github.com/sacloud/service-endpoint-gateway-api-go/releases/tag/v0.1.1) で zones を使ったエンドポイント設定に対応、[v0.1.2](https://github.com/sacloud/service-endpoint-gateway-api-go/releases/tag/v0.1.2) で `NewClient` と List API のバグ修正が入りました。前述の `terraform-provider-sakura` [v3.9.0](https://github.com/sacloud/terraform-provider-sakura/releases/tag/v3.9.0) の Service Endpoint Gateway 対応の土台になっているライブラリです。

### terraform-provider-sakura-policy

旧世代の `terraform-provider-sakuracloud` 向けには、以前から [terraform-provider-sakuracloud-policy](https://github.com/sacloud/terraform-provider-sakuracloud-policy) として、[OPA (Open Policy Agent)](https://www.openpolicyagent.org/) と [Conftest](https://www.conftest.dev/) で使えるポリシールール集が提供されていました。今月は、それの v3 系プロバイダー版である `terraform-provider-sakura-policy` の初版 [v0.1.0](https://github.com/sacloud/terraform-provider-sakura-policy/releases/tag/v0.1.0) がリリースされ、**v3 でも同様にポリシーチェックが使えるようになりました**。

デフォルトで、サービスを安全に運用するための以下のリソースに対するルールが含まれています。

- `sakura_server` / `sakura_disk` / `sakura_vpn_router`
- `sakura_dsr_lb` / `sakura_enhanced_lb` / `sakura_enhanced_db`

たとえば `sakura_server` で **SSH のパスワード認証を有効にしている構成** (`disable_pw_auth = false`) を検出して、公開鍵認証への移行を促してくれるなど、セキュリティ・ガバナンス系のチェックがひととおり入っています。各ルールには [docs.usacloud.jp の解説ページ](https://docs.usacloud.jp/terraform-policy/) も用意されています。Conftest で `conftest pull` してから `conftest test` で評価する形で、ローカル実行も GitHub Actions での CI 組み込みも README に手順が載っています。

もちろんデフォルトルールがすべてのプロジェクトに合うわけではないので、**自分でルールを追加** したり、特定のルールを `exception.json` で **除外** したりといったカスタマイズもできます。

https://github.com/sacloud/terraform-provider-sakura-policy

## おわりに

2026 年 4 月の sacloud OSS は、`terraform-provider-sakura` でのモニタリングスイート対応の充実と、新プロダクトである Service Endpoint Gateway や AppRun 専有型への対応が大きな動きでした。`iam-api-go` での SCIM 対応と認証コンテキスト API の追加も、地味ながら IAM 周りの自動化を考えている方には嬉しいアップデートかと思います。
